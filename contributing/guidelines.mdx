# Contributing Guidelines

We welcome contributions to ZeroQCM. This page covers coding standards, the pull request workflow, and general guidelines.

## Pull Request Workflow

<Steps>
  <Step title="Fork and clone">
    Fork the repository and clone your fork locally.
  </Step>
  <Step title="Create a branch">
    ```bash
    git checkout -b feature/your-feature
    ```
    Use descriptive branch names: `feature/`, `fix/`, `docs/`.
  </Step>
  <Step title="Make changes">
    Make your changes and test them locally with `npm run dev`.
  </Step>
  <Step title="Lint and type check">
    ```bash
    npm run lint
    npm run typecheck
    ```
  </Step>
  <Step title="Commit">
    Use conventional commit messages:
    ```bash
    git commit -m "feat: add bookmark export feature"
    git commit -m "fix: correct quiz session persistence"
    git commit -m "docs: update installation guide"
    ```
  </Step>
  <Step title="Push and open PR">
    ```bash
    git push origin feature/your-feature
    ```
    Open a pull request on GitHub with a clear description of your changes.
  </Step>
</Steps>

## Coding Standards

### TypeScript
- Use TypeScript for all new files
- Define types for props, API responses, and database rows
- Avoid `any` — use proper types or `unknown`

### React Components
- Use functional components with hooks
- Prefer Server Components where possible (App Router)
- Use `"use client"` directive only when client-side interactivity is needed

### Styling
- Use Tailwind CSS utility classes
- Use CSS custom properties from `globals.css` for colors
- Do not hard-code color values in components
- Follow the existing Apple-style aesthetic

### Naming
- **Files**: `kebab-case` for pages and routes, `PascalCase` for components
- **Variables**: `camelCase`
- **Types/Interfaces**: `PascalCase`
- **CSS variables**: `--kebab-case`

## Commit Messages

Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

| Prefix | Use |
|---|---|
| `feat:` | New feature |
| `fix:` | Bug fix |
| `docs:` | Documentation changes |
| `style:` | Code style (no logic change) |
| `refactor:` | Code refactoring |
| `perf:` | Performance improvement |
| `chore:` | Build/tooling changes |

## What to Contribute

- Bug fixes and improvements
- New question sources and faculties
- Accessibility improvements
- Performance optimizations
- Documentation updates
- Translations

<Note>
  For major changes, open an issue first to discuss the approach before starting work.
</Note>
