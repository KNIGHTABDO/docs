# Core Concepts

This page explains the fundamental building blocks of ZeroQCM — how data is organized, how the quiz engine works, and how users interact with the platform.

## Data Hierarchy

ZeroQCM organizes medical questions in a strict hierarchy:

```
Faculty (FMPC, FMPR, FMPM, UM6SS, …)
  └── Semester (S1–S10)
        └── Module (e.g., Anatomie, Biochimie)
              └── Activity (exam session / QCM set)
                    └── Question
                          └── Choice (answer option)
```

### Semesters

Medical studies in Morocco follow a **semester system** (S1 through S10). ZeroQCM covers all semesters:
- **Odd semesters**: S1, S3, S5, S7, S9
- **Even semesters**: S2, S4, S6, S8, S10

Each semester contains modules specific to that year and faculty.

### Modules

A **module** represents a course or subject within a semester (e.g., *Anatomie*, *Biochimie*, *Histologie*). Each module contains multiple **activities** — typically past exam sessions or curated QCM sets.

### Questions and Choices

Each **question** belongs to an activity and has:
- A text body (supports Markdown formatting)
- Multiple **choices** (answer options), each marked as correct or incorrect
- Support for **QCM** (multiple-choice) and **QROC** (open-response) types
- Support for **multi-select** questions with multiple correct answers

## Quiz Flow

The core user journey through ZeroQCM follows this pattern:

<Steps>
  <Step title="Select semester">
    The user picks their current semester from the dashboard or semester browser (`/semestres`).
  </Step>
  <Step title="Choose a module">
    Modules for the selected semester are listed with question counts and progress indicators.
  </Step>
  <Step title="Pick an activity">
    Each module contains one or more activities (exam sessions). The user selects one to start a quiz.
  </Step>
  <Step title="Answer questions">
    Questions are presented one at a time. The user selects one or more options and confirms. Instant feedback shows correct/incorrect answers.
  </Step>
  <Step title="Get AI explanations">
    After answering, the user can tap any option to get an AI-generated explanation of why it is right or wrong.
  </Step>
  <Step title="Review progress">
    The stats page shows overall performance, daily streak, and per-module breakdown.
  </Step>
</Steps>

## Authentication

ZeroQCM uses **Supabase Auth** for user authentication with email/password sign-up. The authentication flow:

1. User signs up or logs in at `/auth`
2. Supabase creates a session with JWT tokens
3. Next.js middleware checks the session on every protected route
4. Unauthenticated users are redirected to `/auth`

### Activation System

After authentication, users must be **activated** before accessing the platform. This is managed through:
- The `activation_keys` table in Supabase
- An admin approval workflow via the admin panel
- Users see `/activate` until their account is approved

### Route Protection

The Next.js middleware protects routes:

| Route Type | Examples | Access |
|---|---|---|
| **Public** | `/`, `/auth` | Everyone |
| **Protected** | `/dashboard`, `/quiz`, `/stats`, `/settings` | Authenticated + activated users |
| **Admin** | `/admin/*` | Admin email only |
| **API** | `/api/*` | Each route handles its own auth |

## AI Explanations

ZeroQCM uses **GitHub Models** for AI-powered explanations. The flow:

1. Student taps an option → frontend sends `{ prompt, model }` to `/api/ai-explain`
2. API route authenticates with a server-side token
3. Response streams back token-by-token for instant perceived performance
4. Explanations are cached in the `ai_explanations` table — all future requests for the same question hit cache instantly

### Model Selection

Users can choose from multiple AI models in the settings page:

## Spaced Repetition

ZeroQCM tracks user performance and uses spaced repetition to improve learning:

- **User answers** are stored in the `user_answers` table
- Questions answered incorrectly 2+ times are flagged as **weak**
- The `/revision` page lists weak modules and launches focused sessions
- Each module shows an **error badge** indicating how many questions were missed

## State Management

ZeroQCM uses **Zustand** for client-side state management and **Supabase** for persistent storage:

| Data | Storage | Persistence |
|---|---|---|
| Quiz session state | `quiz_sessions` table | Per-user, per-activity |
| User answers | `user_answers` table | Permanent |
| AI explanations | `ai_explanations` table | Shared cache |
| Theme preference | `localStorage` | Browser |
| Selected semester | Zustand store | Session |

## Realtime Features

ZeroQCM uses **Supabase Realtime** for collaborative features:

- **Study Rooms**: Real-time collaborative revision sessions where multiple users answer questions together
- **Room Participants**: Live presence tracking of who's in each room
- Rooms use unique codes for joining and have creator-based management

## Design System

ZeroQCM follows an **Apple-style aesthetic** with:

- **CSS custom properties**: Full design token system with no hard-coded colors
- **Dark/Light mode**: System-aware with manual toggle, persisted to `localStorage`
- **Framer Motion**: Staggered entrance animations and spring-based micro-interactions
- **Responsive design**: Optimized for mobile, tablet, and desktop
- **Hairline borders**: Subtle visual separation using `rgba()` borders
